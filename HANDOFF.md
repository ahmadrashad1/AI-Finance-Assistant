## 4. Decisions Made

- **What "unpaid" means lives in `InvoiceService`, not the repository.** `UNPAID_STATUSES = ("sent", "partially_paid", "overdue")` is defined in `domains/finance/services/invoice_service.py`. `InvoiceRepository.list_by_statuses` only knows how to filter by an arbitrary status set — it has no opinion on which statuses count as "unpaid," keeping business meaning out of the data-access layer.
- **`customer_id` is a business code, not the internal UUID.** The LLM never sees schema/PK details, so the tool's `customer_id: str | None` parameter is resolved via `CustomerRepository.get_by_code(...)` (e.g. `"CUST-0007"`), never the raw primary key. An unresolvable `customer_id` raises `ValueError("Customer not found: <id>")`, which `ToolExecutor` already turns into a categorized `status="error"` outcome with no executor changes needed.
- **`days_outstanding` = `max(0, (as_of - due_date).days)`** — days past the due date, zero if not yet due — mirroring the vocabulary the future `get_overdue_invoices(minimum_days=...)` tool will also use, so the two concepts stay aligned.
- **Materiality sort = descending by `balance`** (largest amount owed first) — the plain-English reading of "materiality" for an AR list.
- **`as_of` for `days_outstanding` defaults to real `date.today()`** inside `InvoiceService.get_unpaid_invoices`, overridable via an optional keyword arg for deterministic tests. This is a live, ongoing business query — not the simulator's frozen `SIMULATION_TODAY` clock — so real "today" is correct production behavior; it is *not* exposed as a tool parameter.
- **The `ToolContext(db: AsyncSession)` architecture change was necessary, not incidental.** Before this milestone, `ToolSpec.handler` was `Callable[[Any], Awaitable[BaseModel]]` — params only, no way for a handler to reach the database. `get_unpaid_invoices` is the first DB-backed tool, so `ToolExecutor` now builds a `ToolContext` once per `execute()` call and passes it as a second positional argument to every handler (`get_current_date`'s handler just ignores it). `ToolRegistry`/`get_tool_registry()` remain exactly as DB-free and cacheable as before — this was a one-time, unavoidable ripple across existing tests, done deliberately in Task 1 before any DB-backed tool existed.
- **The Decimal/date JSON-safety fix (`model_dump(mode="json")`) was a required prerequisite, not optional polish** — `get_unpaid_invoices`'s result contains `Decimal`/`date` fields that crash both the `tool_executions` JSONB write and `ChatWorkflow`'s `json.dumps` call under plain `model_dump()`. Fixed once, centrally, in `result_validator.validate_result` before any tool returning non-string data existed.
- **Frontend markdown tables extend the existing hand-rolled renderer, not a new dependency** — consistent with the original Milestone 2 no-dependency choice for `markdown.ts`. No frontend test framework exists in this repo; this was a deliberate, documented scope boundary (verified by script at implementation time, and now confirmed live in a real browser this session).
- Executed directly on `master`, no feature branch — same policy as Milestones 1-4.

## 5. Known Issues / Failing Tests

- **No test/lint/type failures.** All 170 backend tests pass, ruff/mypy are both clean (full project scope), and frontend lint/typecheck/build are all clean, as of this writing.
- **NEW, confirmed live this session — the literal acceptance phrasing fails against full-scale real data with the currently configured LLM.** "Who still owes us money?" against the real reseeded 87-row unpaid-invoices dataset reproducibly gets `413 Payload Too Large` from Groq (`llama-3.1-8b-instant`) at Phase 2, because `ChatWorkflow._build_response_message` (`ai_platform/orchestration/chat_workflow.py`) embeds the tool's *entire* raw JSON result into the Phase-2 prompt with no size cap, truncation, or summarization. Narrower phrasings that resolve to a small result set (tested: 2 rows) work completely, end to end, with the same real LLM and real data. **This is a structural gap, not just a small-model quirk** — any future list-returning AR tool (§7) will hit some version of this against realistic data volumes regardless of provider; a bigger-context model only pushes the limit further out.
- **NEW, confirmed live this session — the `tool_executions` audit-commit-ordering-vs-Phase-2-rollback issue (flagged as theoretical in Milestone 4's HANDOFF) is real and was directly reproduced.** `backend/app/api/chat.py`'s `event_stream()` holds one shared `AsyncSession` for the whole request and only `db.commit()`s after the entire workflow — including Phase-2 streaming — succeeds; any exception (including the 413 above) triggers `db.rollback()`, which erases the `tool_executions` row that `ToolExecutor` already inserted earlier in the same request for a tool call that itself succeeded. Confirmed via direct SQL: 0 `tool_executions` rows after two reproductions of the 413 above, versus a correctly-persisted row when Phase 2 succeeds.
- **`PaymentRepository.record_payment()`'s overpayment/zero/negative-amount validation gap (Milestone 4 HANDOFF §5) is still untouched and still relevant** — no write-tool exists yet, so nothing is broken today, but this must be addressed before any future "record a payment" tool reuses this repository method.
- **`PaymentRepository.record_payment()`'s `as_of` fallback to real `date.today()` instead of `SIMULATION_TODAY`** (Milestone 4 HANDOFF §5) is also still untouched — same reasoning, still only safe because the seed generator always passes `today` explicitly.
- **`ai_platform/evaluation/`** is still README-only — expected, lands in Milestone 8.

## 6. Do NOT Do

- **Don't call a tool handler with only `(params)`** — every handler now takes `(params, context: ToolContext)`, even if it ignores `context` (see `get_current_date`'s handler for the minimal pattern).
- **Don't add a new list-returning tool that dumps its full result into the Phase-2 prompt without a size cap.** §5's live-reproduced 413/rollback failure is the concrete cost of doing this against realistic data volumes — pick a mitigation (top-N + summary counts, pagination, a bigger-context model, or some combination) before shipping `get_overdue_invoices`/`search_invoices`/`list_overdue_customers`.
- **Don't assume the `tool_executions` audit row is durable just because the tool call itself succeeded** — it is not, whenever Phase 2 fails afterward in the same request, until the commit-ordering issue in `backend/app/api/chat.py` is fixed (e.g. committing the tool execution's audit row independently of Phase 2's success/failure).
- Don't reimplement invoice-status derivation logic anywhere else — `compute_invoice_status` in `domains/finance/repositories/invoice_repository.py` remains the one and only source of truth (untouched this milestone).
- Don't reuse `PaymentRepository.record_payment()` behind a real, user-facing tool without first adding the validation from Milestone 4 HANDOFF §5 — still unaddressed, still a must-fix before any write-tool.
- Don't build the `InvoiceAdapter`/`CustomerAdapter` domain-adapter layer prematurely — wait until Milestone 5's tools (now 1 real tool: `get_unpaid_invoices`) reach ≥2, per the existing roadmap (§7 below is where that becomes concrete).
- Don't add `.eslintrc.json` back — ESLint v9 flat config only.
- Don't trust an implementer/reviewer's "pre-existing, unrelated failure" claim without independently reproducing it.
- Don't assume `make` works on this machine without checking.
- Don't push to `origin` without being asked.

## 7. Next Steps (prioritized)

Per `docs/PRD.md`'s roadmap (Chapter 16) and this milestone's own live-verification findings:

1. **Before writing any more list-returning AR tools, address §5/§6's two confirmed gaps**: (a) cap/paginate/summarize tool results before they're embedded in the Phase-2 prompt (`ChatWorkflow._build_response_message`), and (b) fix the `tool_executions` commit-ordering-vs-Phase-2-rollback issue (e.g. commit the audit row in its own transaction, independent of whether Phase 2 later succeeds). Both were only theoretical before this session; both are now directly reproduced against real data.
2. Continue **Domain 1 (Accounts Receivable)**: `get_overdue_invoices`, `get_invoice`, `search_invoices`, `get_customer_balance`, `list_overdue_customers` — registered the same way `get_current_date` (Milestone 3) and `get_unpaid_invoices` (this milestone) were: `ToolSpec` with Pydantic parameter/result models, an async `(params, context: ToolContext)` handler, registered via `ToolRegistry`.
3. Each new AR tool's handler reads through the repositories Milestone 4 built (`InvoiceRepository`, `CustomerRepository`, `PaymentRepository`) plus this milestone's `InvoiceService` pattern for business logic — no new SQL, no schema knowledge leaking into the LLM layer.
4. **Domain Adapters** (PRD Ch.10) — introduce the adapter layer between services and repositories once ≥2 real tools exist. `get_unpaid_invoices` is the first; once `get_overdue_invoices` or another AR tool lands, this becomes concrete and shouldn't be deferred further.
5. Before writing any tool that can record a payment, address `PaymentRepository.record_payment()`'s validation gap and `as_of`/`date.today()` fallback (Milestone 4 HANDOFF §5, still open).
6. Keep the two-phase pipeline, graceful tool-failure degradation, and structured-logging patterns unchanged — real finance tools plug into the exact same `Planner`/`ToolExecutor`/`ChatWorkflow` machinery; the fixes in item 1 should be surgical, not a re-architecture.
7. Once ≥2 tools exist across all domains, revisit Milestone 3's sequential-only tool-execution decision — PRD Ch.13's parallel-execution guidance becomes relevant.
