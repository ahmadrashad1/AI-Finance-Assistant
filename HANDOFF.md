### 2026-07-22/23 session — regression audit, one verified fix, three reverted

**Goal:** complete the case-by-case regression audit the prior session flagged as unverified (§2), and fix anything found, per PRD Ch.24's "any regression on a previously-passing case is a blocking defect" rule.

**Audit result:** confirmed 7 of the 53 legacy cases regressed (listed in §2), plus an 8th, `current_date_basic`, which is now fixed. Each was traced to its actual recorded cassette content, not inferred from the score delta — e.g. `current_date_basic`'s cassette showed the model answering `{"clarification_needed": ...}` for "What's today's date?" despite the prompt explicitly excluding this case.

**What actually shipped — planning prompt v1.5.6, one change:** shape 3 (the `direct_answer` branch) is reordered so its current-date/day-of-week exclusion leads the sentence instead of trailing it (`"NEVER for the current date or day of week - that's always get_current_date (case 2), even though it sounds conversational. Otherwise, ..."`). Verified live at `temperature=0` before and after; confirmed clean (no collateral regressions anywhere else) via a full-suite re-record. Also pins `temperature=0` on the Phase 1 planning call only, in `ai_platform/llm/service.py` (`GroqLLMService.complete`/`AnthropicLLMService.complete` — `stream_reply`, i.e. Phase 2, is untouched). This closes a known, already-deferred item (`docs/MVP-REPORT.md`'s "planning temperature ≈0" post-MVP priority) and is what made the rest of this session's methodology possible: every prior recording pass was one non-deterministic sample, so a single-case live probe could not previously distinguish "the fix worked" from "got a different random sample."

**Three other fixes were attempted, verified live, and reverted — this is the important finding for future work on this prompt:**
1. A bare-fragment-rule rewrite, meant to stop "Anchor Components"/"Pioneer Manufacturing" from being misread as ambiguous fragments (the rule's own example, "credit exposure for Anchor," collides by substring with the real customer "Anchor Components"). Verified: it did stop the spurious clarification, but a full-suite re-record showed it introduced a *new*, worse bug — `get_customer_balance` calls started resolving to `get_vendor_balance` instead, across several previously-clean cases (`customer_balance_granite`, `hallucination_customer_not_found`, `explanation_quality_customer_balance`). Best working theory: the rewrite's example list put a customer name and a vendor name side by side as undifferentiated "full name" examples, and the model picked up an unintended association.
2. A `get_aging_report` vs. `get_vendor_invoices` disambiguation clause. Verified: it fixed `vendor_invoices_overdue_status`, but the same full-suite re-record showed it broke `aging_report_basic`, which had been passing.
3. A `find_duplicate_invoices` paraphrase addition ("has INV-nnnnn been entered/submitted before?"). Verified: it fixed `duplicate_detection_none_found`, but broke three sibling duplicate-detection cases that had been passing (`duplicate_detection_broad_scan`, `duplicate_detection_second_pair`, `duplicate_detection_specific_invoice_found`, plus `explanation_quality_duplicate_pair`).
4. (A fourth, `resolve_date_range`'s tool description gaining an explicit "reject explicit calendar dates" clause, for `search_invoices_due_before_and_min_amount`.) This one didn't even achieve its own goal on a clean re-record — the model kept calling `resolve_date_range` on "June 1, 2025" regardless. An earlier isolated probe that looked like a pass was later found to have been invalidated by a same-minute 413 on the very next diagnostic call (Groq's per-minute token budget), not a genuine confirmation — a caution for this session's own methodology, corrected before it was trusted further.

All four attempts were reverted to their exact pre-session wording (byte-diffed against the committed file to confirm) rather than shipped, since none held up under a full-suite check. **The takeaway carried forward: for this model at this tool-catalog size (26 tools), a single-case live probe at temperature=0 can confirm a fix does what it's meant to, but cannot rule out collateral damage elsewhere — only a full-suite re-record can.** That makes iterating on this prompt expensive (see the infrastructure note below), and is itself evidence for pulling PRD Ch.24's domain-routing mitigation (Milestone 14) forward rather than continuing to hand-tune one monolithic 26-tool prompt.

**Infrastructure finding, unrelated to the prompt itself:** getting one full, clean 89-case live recording pass this session required **17 separate relaunches over several hours**, because the sandbox environment kills long-running background processes at an unpredictable point (observed anywhere from under a minute to ~35 minutes), independent of the `timeout` parameter passed to the launching tool. Recovered each time using the cassette skip-logic (item 7 above) to resume rather than restart. Also discovered: rapid back-to-back manual diagnostic calls (outside the harness's own 100-second throttle) can collide with Groq's rolling per-minute token window and produce a misleading 413, even when the actual prompt is well under budget — this cost real time before being identified as measurement noise rather than a real budget problem. And once: a single case (`assess_risk_should_increase`) scored as a false regression that turned out to be a transient DB-state artifact from an earlier reseed mid-session, not a prompt or model issue — resolved by reseeding again immediately before the final recorded-mode pass. None of this is prompt-specific; it's recorded here so a future session doesn't have to rediscover it.

**Cassette bookkeeping:** the 89-case suite's cassettes now live under hash `910e5564d192` (`sha256("1.5.6:1.5.1")[:12]`, since `PLANNING_PROMPT_VERSION` bumped but `SYSTEM_PROMPT_VERSION` did not change). 89 of the 91 files (all except `current_date_basic`/`current_date_day_of_week`, whose behavior is genuinely different now) are byte-identical copies of the prior session's committed `00191e7157d5` cassettes — confirmed safe to copy rather than re-record because the underlying prompt text those cases depend on is, after the reverts above, byte-identical to what produced the `00191e7157d5` recordings. `followup_total_for_granite`'s second turn remains uncassetted (documented pre-existing infrastructure limitation, unchanged — see the original session's §5/§6 below).

## 5. Decisions Made

- **`get_expense_claims.claim_number` was added beyond the PRD's literal tool signature** (Task 2's design note) — a claim-number exact-lookup is the same shape of request as `search_invoices`' `invoice_number`, and omitting it would have been an inconsistent gap in an otherwise-symmetric API.
- **`payment_prioritization.yaml`'s expected-tools update is an intentional capability upgrade, not a regression** (Task 8's design note) — "which invoices should I pay first" now routes to `get_payment_prioritization` alone, replacing the pre-M12 `get_vendor_invoices`+`get_cash_position` combo that was the only mechanism available before this milestone.
- **`CashFlowService` has two documented deterministic adjustment rules** (Task 6's design note) — see the service's own docstrings for the exact rules; called out here per CLAUDE.md's Phase 2 Step 5 ("record non-obvious design decisions").
- **`assess_credit_risk` returns evidence only, never a recommendation** — verified twice: once via `AssessCreditRiskResult`'s field list (no `recommendation`-shaped field exists) during Task 10's review, and again during Task 13's manual UI verification by re-reading the model's actual response, which reasoned over the evidence itself rather than echoing a canned verdict.
- **Groq's real constraint on this account is a daily token cap (TPD), not just the per-minute cap (TPM)** — root-caused mid-Task-12 via a direct diagnostic call reading the raw Groq error body (`... on tokens per day (TPD): Limit 500000, Used 496529 ...`), not visible from the TPM-only rate-limit headers alone. A second Groq account (separate org, independent daily budget) was used to finish the last stretch of recording once the first account's daily budget was exhausted for the day.
- **One case's cassette (`followup_total_for_granite`, second turn) could not be recorded under any key** — its conversation history pushes the Phase 1 planning call to 6,669 tokens against Groq's fixed 6,000 TPM *per-request* hard cap (a 413, non-retryable by definition — this is a single-request size limit, not a budget that recovers with time or a fresh key). Reproduced directly against the real workflow to confirm this exact root cause before accepting it as a documented infrastructure limitation rather than a prompt defect.
- **Prompt version bumps were kept to two (v1.5.4, v1.5.5) rather than many small increments**, each requiring a full cassette re-record under this account's tight budget — batching fixes before bumping VERSION was adopted mid-session specifically to reduce how many full-suite re-records this constraint forces (see §6).

## 7. Do NOT Do

- All Milestone-11 rules still stand (rate-limited output isn't model behavior, don't bump prompt VERSIONs casually, keep eval case ids ≤44 chars, don't push without being asked, don't run pytest between reseeding and anything reading seeded data, don't hardcode expected anomaly values, don't change simulator v1 constants).
- **Don't add Phase B/C domain tools claiming they're "Milestone 12 work"** — Budgets/FP&A, Bank Reconciliation, Fixed Assets are Milestone 13 (Phase B), per `docs/PRD.md` Ch.27.
- **Don't route "which invoices should I pay first" back through the old `get_vendor_invoices`+`get_cash_position` combo** — `get_payment_prioritization` now owns that phrasing exclusively (§5).
- **Don't run a full live `--record` pass without first checking this Groq account's remaining daily budget** (a single diagnostic call reading the raw 413 error body, not just the TPM headers) — a full 89-case/92-turn re-record costs roughly 90-100 live calls and can exhaust the daily 500,000-token cap on its own; confirm budget before starting, not after several failed attempts.
- **Don't bump `PLANNING_PROMPT_VERSION`/`SYSTEM_PROMPT_VERSION` for a small wording tweak without batching it with other pending fixes** — every version bump invalidates every cassette (by design, since the hash is version-derived) and forces a full suite re-record under this account's tight budget; two bumps in one session (v1.5.4, v1.5.5) each cost the better part of a day this time.
- **Don't assume `/api/chat` will succeed on the first try right now** — see §6's rate-limit-backoff gap; space manual test queries roughly 70-90s apart with zero other Groq calls in between, or expect a generic "I couldn't process that right now" on the first attempt.
