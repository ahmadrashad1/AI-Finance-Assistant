## 2. Work Completed This Session

This milestone shipped, in order, across a design spec and a 22-task implementation plan:

- **Data model & case loading (Tasks 1-4)**: three new tables in the already-existing (but empty) `evaluation` Postgres schema ‚Äî `evaluation_cases` (a queryable mirror of each YAML case, upserted by business `case_id`, not a second source of truth), `evaluation_runs` (one row per CLI invocation, carrying both prompt VERSIONs), `evaluation_results` (one row per case per run: expected vs. actual JSON, score, per-check metrics, failure reason). `EvaluationRepository` wraps all three. `EvalCase`/`Expectations`/`ExpectedTool` (Pydantic) define the case-file schema, with a validator rejecting a case that sets both `expected_clarification` and `expected_tools` (mutually exclusive by design). `load_suite()` reads YAML files from an injectable `evals_root` (defaults to the real repo `evals/` directory, so no test ever writes into it).
- **Cassette mechanism (Tasks 5-6)**: `prompt_version_hash()` hashes both prompt `VERSION`s together; a cassette's filename embeds it, so bumping either prompt makes every existing cassette go stale ‚Äî the entire mechanism behind "prompt changes without a passing eval run are flagged," with no separate bookkeeping. `ScriptedLLMService` (replays a cassette in `recorded` mode) and `RecordingLLMService` (wraps the real `LLMService` in `live`/`record` mode, buffering what it returns) are the third and fourth production `LLMService` implementations alongside `AnthropicLLMService`/`GroqLLMService` ‚Äî deliberately distinct from `backend/tests/fakes.py`'s test-only `FakeLLMService`. Both expose `stream_reply_called`, letting the runner distinguish a clarification turn (Phase 2 never runs) from a normal one without inspecting `ChatEvent` shapes, which are structurally ambiguous between the two.
- **Scoring (Tasks 7-8)**: `score_case` runs five independent checks (tool-name sequence, parameter subset match ‚Äî with a `<piped>` sentinel that accepts any resolved, non-placeholder value for parameters whose real value is unpredictable ‚Äî clarification bool-or-regex match, forbidden-content/hallucination check, required-facts check), each with its own dedicated failing-path test. `aggregate_metrics` computes the four named suite-level metrics, each restricted to the cases where it's actually applicable (an empty denominator defaults to `1.0`, never divides by zero).
- **The runner (Tasks 9-11)**: `ToolExecutionRepository.list_by_request_id` (new method) lets the runner reconstruct a turn's real, resolved tool calls from `application.tool_executions` ‚Äî the same audit trail production code already writes ‚Äî rather than adding a field to `ChatEvent`. `EvaluationRunner.run_case` builds a real `ChatWorkflow` per conversation turn (real `Planner`, `ExecutionPlanner`, `ToolExecutor`, `PromptBuilder`, mirroring `app/api/chat.py`'s own wiring exactly), swapping only the `LLMService`. **Zero lines changed in `chat_workflow.py`, `planner.py`, `execution_planner.py`, or `tool_registry/executor.py`** ‚Äî the whole framework is a read-only, bolt-on consumer of existing production code. A missing cassette raises `CaseStale`. Task 11's deeper integration tests (live/record cassette round-trip, a hand-authored two-cassette follow-up case) caught a genuine bug in Task 10's own just-shipped code: `request_id` was fully deterministic (`f"eval-{case_id}-turn{turn}"`), so two `run_case` invocations for the same case (exactly what re-running `--record` on an already-recorded suite does) would have `list_by_request_id` return tool-execution rows from *both* invocations ‚Äî fixed with a fresh `uuid4`-derived token per `run_case` call, scoped entirely to `runner.py`.
- **Report and CLI (Tasks 12-14)**: `render_scorecard` (plain string formatting, no new dependency) renders the per-category breakdown, the four metrics, and a STALE section. `run.py`'s `run_suite`/`main` tie everything together: `--record` unconditionally forces `mode="live"` regardless of `--mode`; the command exits non-zero if any case fails **or** is STALE. A new `evaluation` job in `.github/workflows/ci.yml` runs the deterministic subset on every PR with no `LLM_API_KEY` needed.

## 4. Decisions Made

- **LLM-response cassettes, not hand-scripted mocks, are the CI determinism mechanism.** A hand-written `plan_response` (the pattern `test_chat_eval.py` already used for Milestone 7's two eval tests) is trivially deterministic but makes tool-selection accuracy pass by construction ‚Äî it cannot detect a real NLU/prompt regression, which is this milestone's actual purpose. A cassette is captured from one real model call and replayed; its filename embeds a hash of both prompt `VERSION`s, so a prompt change alone (no re-recording) makes every cassette stop matching. This single mechanism does double duty: CI determinism, and the "prompt changes without a passing eval run are flagged" requirement.
- **Actual tool calls are read from `application.tool_executions`, not added to `ChatEvent`.** `ChatEvent` only ever carried `{type, content, conversation_id, message, tool}` ‚Äî no parameters. Rather than widen a production dataclass for evaluation's benefit, the runner queries the same audit trail `ToolExecutor` already writes for every real request, via a new `request_id`-scoped repository method. Zero production classes changed as a result.
- **`ScriptedLLMService`/`RecordingLLMService` are production code, not test doubles**, and deliberately do not import or reuse `backend/tests/fakes.py`'s `FakeLLMService` ‚Äî they're the third/fourth real implementation of the `LLMService` protocol (used by `run.py` in every mode), architecturally parallel to `AnthropicLLMService`/`GroqLLMService`, not test infrastructure.
- **The numeric-string parameter-matching leniency is narrow and explicit**, not a general fuzzy-match relaxation: only applies when the *expected* value is a plain `int`/`float` (never `bool` ‚Äî guarded explicitly against Python's `isinstance(True, int)` being `True`), only when the actual value is a string that parses to the same number via `Decimal`. String-vs-string exact-match (names, invoice numbers, dates) and the `<piped>` sentinel path are both completely untouched.
- **A single case's unhandled exception must never crash the whole suite run** ‚Äî mirrors the existing `CaseStale` handling exactly (persist a failed result, `continue` to the next case), discovered as a real gap only once the framework was actually pointed at a real, occasionally-malformed LLM response.
- **Genuine model-behavior gaps found during recording are documented, not hidden by loosening case expectations.** Two case files had real authoring bugs (an assumed-but-nonexistent tool parameter, a customer/vendor name collision) and were fixed, since the *case* was wrong. The other 10 failing cases are left failing, because the *model's current behavior* is what's actually being measured ‚Äî see ¬ß5.
- Executed directly on `master`, no feature branch ‚Äî reconfirmed with the user at the start of this session's implementation phase (same policy as Milestones 1-7).

## 6. Do NOT Do

- **Don't hand-script cassette content instead of recording it against a real model.** A human-authored `plan_response` makes tool-selection accuracy pass by construction and cannot detect a real prompt/NLU regression ‚Äî the entire point of this milestone. Use `--record`.
- **Don't add a `parameters` field to `ChatEvent` for evaluation's benefit.** Read actual tool calls from `application.tool_executions` via `ToolExecutionRepository.list_by_request_id` instead ‚Äî this milestone's entire framework is a bolt-on, read-only consumer of existing production code, and should stay that way.
- **Don't bump `planning_prompt.VERSION` or `system_prompt.VERSION` as part of routine eval maintenance.** That's exactly what should make every existing cassette go stale ‚Äî it's the mechanism, not an accident to avoid. A genuine prompt content change *should* require re-recording (`--record`) before the eval suite goes green again.
- **Don't run the full pytest suite (or anything using `clean_db`) between reseeding and a cassette-recording run.** `clean_db` truncates the same finance tables the recording run needs real data from ‚Äî this bit this exact session (see ¬ß2 item 4) and produced a batch of misleading "failures" that were actually just empty-database artifacts. Reseed immediately before recording, with nothing intervening.
- **Don't loosen an eval case's expectations just to make a genuinely-failing case pass.** Distinguish a real case-authoring bug (fix it ‚Äî two examples this session: an assumed-but-nonexistent tool parameter, a customer/vendor name collision) from a real model-behavior gap (leave it failing, document it ‚Äî ten examples in ¬ß5). Masking the second kind defeats the framework's purpose.
- **Don't assume the numeric-parameter comparison leniency applies broadly.** It's narrowly scoped to plain `int`/`float` expected values compared against a numeric-looking actual string ‚Äî never relaxes string-vs-string exact matches or the `<piped>` sentinel path.
- **Don't assume a case with `required_facts` naming a specific entity will reliably appear in a broad, many-row list result's prose response.** The model correctly truncates large result sets (e.g. 87 overdue invoices) to a top-N-by-amount table with a "showing X of Y" summary ‚Äî a specific example entity may not make that cutoff. Reserve `required_facts` for flat/single-value results or narrow, single-row list results.
- **Don't assume Docker Desktop is running** ‚Äî check (`docker compose ps`) and start it if needed before any DB-dependent work.
- **Don't trust an implementer/reviewer's "pre-existing, unrelated failure" claim without independently reproducing it** (still true ‚Äî this session reconfirmed a "171 mypy errors" claim from one fix subagent was noise from their own diagnostic process, not a real regression, by re-running mypy directly).
- **Don't push to `origin` without being asked.**
