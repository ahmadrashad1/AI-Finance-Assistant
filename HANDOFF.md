## 5. Decisions Made

- **`get_expense_claims.claim_number` was added beyond the PRD's literal tool signature** (Task 2's design note) — a claim-number exact-lookup is the same shape of request as `search_invoices`' `invoice_number`, and omitting it would have been an inconsistent gap in an otherwise-symmetric API.
- **`payment_prioritization.yaml`'s expected-tools update is an intentional capability upgrade, not a regression** (Task 8's design note) — "which invoices should I pay first" now routes to `get_payment_prioritization` alone, replacing the pre-M12 `get_vendor_invoices`+`get_cash_position` combo that was the only mechanism available before this milestone.
- **`CashFlowService` has two documented deterministic adjustment rules** (Task 6's design note) — see the service's own docstrings for the exact rules; called out here per CLAUDE.md's Phase 2 Step 5 ("record non-obvious design decisions").
- **`assess_credit_risk` returns evidence only, never a recommendation** — verified twice: once via `AssessCreditRiskResult`'s field list (no `recommendation`-shaped field exists) during Task 10's review, and again during Task 13's manual UI verification by re-reading the model's actual response, which reasoned over the evidence itself rather than echoing a canned verdict.
- **Groq's real constraint on this account is a daily token cap (TPD), not just the per-minute cap (TPM)** — root-caused mid-Task-12 via a direct diagnostic call reading the raw Groq error body (`... on tokens per day (TPD): Limit 500000, Used 496529 ...`), not visible from the TPM-only rate-limit headers alone. A second Groq account (separate org, independent daily budget) was used to finish the last stretch of recording once the first account's daily budget was exhausted for the day.
- **One case's cassette (`followup_total_for_granite`, second turn) could not be recorded under any key** — its conversation history pushes the Phase 1 planning call to 6,669 tokens against Groq's fixed 6,000 TPM *per-request* hard cap (a 413, non-retryable by definition — this is a single-request size limit, not a budget that recovers with time or a fresh key). Reproduced directly against the real workflow to confirm this exact root cause before accepting it as a documented infrastructure limitation rather than a prompt defect.
- **Prompt version bumps were kept to two (v1.5.4, v1.5.5) rather than many small increments**, each requiring a full cassette re-record under this account's tight budget — batching fixes before bumping VERSION was adopted mid-session specifically to reduce how many full-suite re-records this constraint forces (see §6).

## 7. Do NOT Do

- All Milestone-11 rules still stand (rate-limited output isn't model behavior, don't bump prompt VERSIONs casually, keep eval case ids ≤44 chars, don't push without being asked, don't run pytest between reseeding and anything reading seeded data, don't hardcode expected anomaly values, don't change simulator v1 constants).
- **Don't add Phase B/C domain tools claiming they're "Milestone 12 work"** — Budgets/FP&A, Bank Reconciliation, Fixed Assets are Milestone 13 (Phase B), per `docs/PRD.md` Ch.27.
- **Don't route "which invoices should I pay first" back through the old `get_vendor_invoices`+`get_cash_position` combo** — `get_payment_prioritization` now owns that phrasing exclusively (§5).
- **Don't run a full live `--record` pass without first checking this Groq account's remaining daily budget** (a single diagnostic call reading the raw 413 error body, not just the TPM headers) — a full 89-case/92-turn re-record costs roughly 90-100 live calls and can exhaust the daily 500,000-token cap on its own; confirm budget before starting, not after several failed attempts.
- **Don't bump `PLANNING_PROMPT_VERSION`/`SYSTEM_PROMPT_VERSION` for a small wording tweak without batching it with other pending fixes** — every version bump invalidates every cassette (by design, since the hash is version-derived) and forces a full suite re-record under this account's tight budget; two bumps in one session (v1.5.4, v1.5.5) each cost the better part of a day this time.
- **Don't assume `/api/chat` will succeed on the first try right now** — see §6's rate-limit-backoff gap; space manual test queries roughly 70-90s apart with zero other Groq calls in between, or expect a generic "I couldn't process that right now" on the first attempt.
