## 4. Decisions Made

- **Prerequisite fixes came first (Tasks 1-2), before any new tool** — every tool in this milestone returns a list, and all four would reproduce Milestone 5's live-reproduced 413 at realistic data scale, so both fixes landed first and are confirmed (not just theorized) working before the four vertical slices were built on top of them.
- **Payload capping shapes only what Phase 2 sees, never what the tool returns or what gets audited.** `cap_result_for_prompt` in `ai_platform/orchestration/result_shaping.py` truncates list-valued fields to the first 10 items only when building the Phase-2 prompt string; `tool_executions` and the tool's actual return value are always the full, uncapped result. Every tool already sorts by materiality or urgency, so "first 10" means "the 10 most important rows," and a `summary` block (never a list) still carries the true aggregate even when truncated.
- **Audit durability is achieved by committing the same shared session immediately inside `ToolExecutor.execute()`, not by adding a second `AsyncSession`.** A second session was considered and rejected because of the `tool_executions.conversation_id` → `conversations.id` FK: a brand-new conversation's row is only `flush()`-ed (not committed) before the tool loop runs, so a separate session couldn't see it yet and would raise a FK violation on every first-turn tool call.
- **`search_invoices`'s `minimum_amount`/`maximum_amount` filter on `total`, not `balance`** — "amount" in a search context is naturally the invoice's face amount, a deliberately different semantic from Milestone 5's `get_unpaid_invoices.minimum_amount`, which filters on outstanding *balance*.
- **`get_overdue_invoices` reuses `InvoiceRepository.list_by_statuses` with the strict `statuses=("overdue",)`** rather than Milestone 5's broader `UNPAID_STATUSES` — "overdue" is deliberately narrower than "unpaid," and the two tools sort differently (urgency vs. materiality) to match what each question is really asking.
- **`get_customer_balance`/`get_vendor_balance` resolve by company name, not business code**, via new `CustomerRepository.get_by_name`/`VendorRepository.get_by_name` (case-insensitive exact match) — PRD Ch.10 specifies these tools by name, and a user says "Northwind Manufacturing," never "CUST-0001." No tool in this system hands the LLM a name→code lookup.
- **`get_vendor_balance`'s `OUTSTANDING_PO_STATUSES = ("approved", "received")`** — `draft` purchase orders aren't yet a commitment, `cancelled` ones never were.
- **No frontend changes this milestone** — every new tool's result is either `{invoices: [...], summary: {...}}`-shaped (reusing Milestone 5's GFM markdown-table renderer) or a flat record (`get_customer_balance`/`get_vendor_balance`, rendered as plain prose by the existing system prompt instruction), so no new rendering logic was needed. This session's live browser check exercised both shapes and confirmed both render correctly with zero frontend changes.
- **AI evaluation tests continue Milestone 5's established pattern exactly**: `FakeLLMService(plan_response=...)` hardcoded per test, parametrized over phrasings, asserting the resulting `tool_call` event matches. A full evaluation *framework* (real/recorded LLM graded on tool-selection accuracy) remains Milestone 8 scope.
- **Planning-prompt disambiguation lives in the prompt, never in code** — the `get_unpaid_invoices` vs. `get_overdue_invoices` overlap is resolved by an explicit rule in `planning_prompt.py`'s Rules section (Task 15), keeping tool selection entirely in the LLM/prompt layer per CLAUDE.md's "no keyword matching anywhere" rule.
- Executed directly on `master`, no feature branch — same policy as Milestones 1-5.

## 6. Do NOT Do

- **Don't call a tool handler with only `(params)`** — every handler takes `(params, context: ToolContext)`, even if it ignores `context`.
- **Don't add a new list-returning tool that dumps its full result into the Phase-2 prompt without going through `cap_result_for_prompt`.** This milestone's Task 1 fix, and this session's live re-confirmation that it resolves Milestone 5's 413 finding, is the concrete proof this matters — any new list-returning tool must route through `ai_platform/orchestration/result_shaping.py`, not bypass it.
- **Don't assume a tool's `description` field reaches Phase 2.** §5's `get_vendor_balance` finding is the concrete cost of assuming this — Phase 1 (planning) reads `ToolSpec.description`; Phase 2 (`ChatWorkflow._build_response_message`) only ever sees `{tool, status, result, error}`. Any caveat the user must see has to live in the tool's *result* (or the system prompt), not just its description.
- **Don't add a second `AsyncSession` to make `tool_executions` durable.** The FK from `tool_executions.conversation_id` to a possibly-still-`flush()`-only `conversations` row is why Task 2 committed the *same* shared session immediately inside `ToolExecutor.execute()` instead — see §4.
- **Don't reimplement invoice-status derivation logic anywhere else** — `compute_invoice_status` in `domains/finance/repositories/invoice_repository.py` remains the one and only source of truth (untouched this milestone).
- **Don't reuse `PaymentRepository.record_payment()` behind a real, user-facing tool without first adding the validation from Milestone 4 HANDOFF §5** — still unaddressed, still a must-fix before any write-tool.
- **Don't defer Domain Adapters again** — per Milestone 5 HANDOFF §7 item 4, the adapter layer was to become concrete once ≥2 real tools existed; there are now **5 real finance-domain tools** (`get_unpaid_invoices`, `search_invoices`, `get_overdue_invoices`, `get_customer_balance`, `get_vendor_balance`), plus `get_current_date`. This is now genuinely due, not premature (§7).
- **Don't run the full pytest suite and then assume the seeded dev database is still populated** — `backend/tests/conftest.py`'s `clean_db` fixture truncates the same `DATABASE_URL` the app uses in dev, as confirmed live this session. Reseed after running tests, before any manual/live check that needs real data.
- **Don't add `.eslintrc.json` back** — ESLint v9 flat config only.
- **Don't trust an implementer/reviewer's "pre-existing, unrelated failure" claim without independently reproducing it.**
- **Don't assume `make` works on this machine without checking.**
- **Don't push to `origin` without being asked.**
