## 4. Decisions Made

- **`SIMULATION_TODAY = date(2026, 7, 8)` is a fixed constant** in `domains/finance/simulator/constants.py` — `datetime.now()`/`date.today()` is never called anywhere in the `domains/finance/simulator/` package. This is required for the design spec's "same seed → same data" guarantee to hold indefinitely (not just on the day it was first seeded) — verified directly by Task 13's repeatability test, which truncates and reseeds twice and asserts identical business-data content. `consistency_check.py` uses the same constant (not the real calendar date) when judging whether an invoice's due date has passed, so the check's verdict is stable regardless of when it's run.
- **Internal database primary keys (`uuid.uuid4()`) are explicitly EXEMPT from the determinism contract** — decided during Task 9's code review. UUIDs are opaque internal identifiers, never user-facing (per PRD Ch.12). The actual reproducibility guarantee (Task 13's test) is scoped to business-data content — row counts, invoiced totals by customer code, etc. — not PK equality across reseeds. **Do not** let a future milestone "fix" this by trying to make UUIDs seeded/deterministic — that would be an inconsistent, unrequested change; nothing in the design spec or PRD asks for it, and nothing downstream depends on PK stability across reseeds.
- **`compute_invoice_status(*, total, amount_paid, due_date, as_of, current_status) -> str`** is THE one real piece of business logic in this milestone, and is the single source of truth for invoice-status derivation — don't reimplement this logic elsewhere. The priority rule: `cancelled`/`draft` are preserved as manually-controlled states, never derived; otherwise, in order, `paid` (balance ≤ 0) beats `overdue` (due_date < as_of) beats `partially_paid` (amount_paid > 0) beats `sent`. Notably, **overdue explicitly beats partially_paid** — a partially-paid invoice that's also past its due date is `overdue`, not `partially_paid`. Called identically by both `PaymentRepository.record_payment()` and the seed generator.
- **`PaymentRepository.record_payment()` has no guard against overpayment, zero, or negative amounts** — a deliberate decision, not a gap. CLAUDE.md's "don't validate scenarios that can't happen" applies here: this milestone's only caller (the seed generator) only ever passes valid, well-formed amounts by construction. **Flagged in §5 as a must-fix for Milestone 5+**: a real "record a payment" tool will have genuine user/API input and MUST add proper validation (amount > 0, amount ≤ outstanding balance, etc.) before this repository method is reused behind a real tool.
- **Explicitly out of scope this milestone**, matching the design spec's Scope Boundary — don't let future work assume these exist:
  - The `InvoiceAdapter`/`CustomerAdapter` domain-adapter layer — comes after Milestone 5's real tools exist (see old roadmap item 5, still pending, now §7 item 3 below).
  - Persona classes / named scenario packs — this milestone is one static, deterministically-generated company ("Northwind Manufacturing Ltd."), not a library of multiple named datasets. The per-customer `BEHAVIOR_WEIGHTS` bucket (`reliable`/`average`/`slow`/`risky`) in `constants.py` biases payment timing/coverage per customer but is *not* a formal Persona system.
  - Soft deletes — none of the 11 models have a `deleted_at`/`is_deleted` column.
  - An `events` table — not built this milestone.
- Executed directly on `master`, no feature branch — same policy as Milestones 1-3.
- Design spec and implementation plan for this milestone, for reference: `docs/superpowers/specs/2026-07-08-milestone-4-finance-simulator-design.md` and `docs/superpowers/plans/2026-07-08-milestone-4-finance-simulator.md`.

## 5. Known Issues / Failing Tests

- **None new.** All 141 backend tests pass, ruff/mypy are both clean as of this writing (full project scope: `app alembic ../ai_platform ../domains`).
- **`PaymentRepository.record_payment()`'s overpayment/zero/negative-amount validation gap (§4) is a must-fix for Milestone 5+, not this milestone.** The seed generator is the only current caller and only ever passes valid amounts by construction, so nothing is broken today — but a real "record a payment" tool built on top of this repository method in Milestone 5 or later MUST add that validation before exposing it to real (LLM-planned, ultimately user-driven) input. Don't reuse `record_payment` behind a real tool without addressing this first.
- **`PaymentRepository.record_payment()`'s `as_of` computation falls back to real `date.today()`, not `SIMULATION_TODAY`, when its optional `today` parameter is omitted** (`as_of = today if today is not None else date.today()`). This is fine today because every current caller (the seed generator) always passes `today=SIMULATION_TODAY` explicitly — the fallback path is never exercised. It lives in `domains/finance/repositories/`, not the `domains/finance/simulator/` package, so it isn't a literal violation of the "no `date.today()` in the simulator package" rule (§4) — but a future Milestone 5+ tool calling `record_payment` without passing `today` would silently get the real calendar date instead of `SIMULATION_TODAY`, breaking determinism for anything built on the simulator's fixed-anchor contract. Worth a decision (e.g. defaulting to `SIMULATION_TODAY` instead of `date.today()`, or requiring callers to always pass it explicitly) before Milestone 5 builds a real payment-recording tool on top of this.
- **`ai_platform/evaluation/`** is still README-only — expected, lands in Milestone 8. Unchanged from Milestone 3.
- All Known Issues carried over from Milestone 3's HANDOFF (the `ChatWorkflow.log()` post-`finally` ContextVar reset, the `tool_executions` audit-commit-ordering-vs-Phase-2-rollback issue, the unconstrained `tool_executions.status` column) are **unchanged and untouched this milestone** — this milestone did no work in `ai_platform/` or `backend/app/`. ## 6. Do NOT Do

- Don't call `datetime.now()`/`date.today()` anywhere in `domains/finance/simulator/` — use `SIMULATION_TODAY` from `constants.py`. This is what makes "same seed → same data" hold indefinitely, not just today.
- Don't try to make `uuid.uuid4()` primary keys seeded/deterministic — internal PKs are explicitly exempt from the determinism contract (§4). The reproducibility guarantee is about business-data content, not PK equality.
- Don't reimplement invoice-status derivation logic anywhere else — `compute_invoice_status` in `domains/finance/repositories/invoice_repository.py` is the one and only source of truth, and both current callers (`PaymentRepository.record_payment`, the seed generator) already use it identically.
- Don't reuse `PaymentRepository.record_payment()` behind a real, user-facing tool without first adding overpayment/zero/negative-amount validation (§5) — the current lack of validation is only safe because the seed generator is a controlled, well-formed-input-only caller.
- Don't build the `InvoiceAdapter`/`CustomerAdapter` domain-adapter layer, persona/scenario-pack classes, soft deletes, or an `events` table this milestone claimed to need — all four are explicitly out of scope per the design spec's Scope Boundary (§4). The adapter layer specifically should wait until Milestone 5's real tools exist, per the existing roadmap.
- Don't add real finance tools directly on top of the simulator without going through `ToolRegistry`/`ToolSpec` — Milestone 5 must register Accounts Receivable tools the exact same way `get_current_date` was registered in Milestone 3.
- Don't add `.eslintrc.json` back — ESLint v9 flat config only (carried over from Milestone 3, frontend untouched this milestone).
- Don't trust an implementer/reviewer's "pre-existing, unrelated failure" claim without independently reproducing it (carried over from Milestone 3's hard-won lesson).
- Don't assume `make` works on this machine without checking.
- Don't push to `origin` without being asked.
