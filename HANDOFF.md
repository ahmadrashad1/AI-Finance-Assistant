## 4. Decisions Made

- **The planner does NOT receive today's date as a given fact in its context** тАФ a deliberate deviation from PRD Ch.13's "Build AI Context" step, which lists "Current Date" as a fixed context element. Injecting it here would let the model answer "what's today's date" from context and never call `get_current_date()`, defeating the one thing this milestone exists to prove. Revisit only if a real "as of" date reference need emerges elsewhere (e.g. a "days overdue" tool) тАФ that's a separate concern from this milestone's planner context.
- **Phase 1 uses strict JSON-mode prompting, not native provider tool-calling APIs** тАФ one uniform, provider-agnostic parsing/validation path (a `Plan` Pydantic model with an exactly-one-branch validator) rather than two different native tool-calling schemas that still wouldn't express the clarification/direct-answer branches natively. Groq's `response_format=json_object` is used where available; Anthropic relies on prompt instructions only.
- **A failed tool call degrades gracefully, it never crashes the request** тАФ unknown tool name, bad parameters, handler exception, and malformed result all become a `status="error"` outcome that's still persisted and still fed to Phase 2 so the LLM can explain the failure (PRD Ch.13 Error Recovery Branch), rather than raising and killing the turn.
- **Sequential-only tool execution, no parallel execution graph** тАФ there is exactly one registrable tool this milestone, so a concurrency dispatcher would be pure speculation. Revisit once тЙе2 independently-selectable tools exist.
- **`tool_call` SSE event added to the wire contract** (not required by the stated acceptance criteria, which only needed DB/log visibility) тАФ an explicit user choice, made during design brainstorming, to also surface tool activity in the UI rather than keep it backend-only.
- **`get_current_date` lives in `ai_platform/tool_registry/tools/`, not `domains/finance/tools/`** тАФ it's a generic infrastructure-proving tool, not a finance capability. `domains/finance/` stays README-only until Milestone 4/5.
- Executed directly on `master`, no feature branch тАФ same policy as Milestones 1-2.

## 5. Known Issues / Failing Tests

- **None.** All 99 backend tests pass, ruff/mypy/ESLint/tsc/`next build` are all clean as of this writing.
- **A real, subtle bug was caught mid-milestone by independently reproducing rather than trusting an implementer's "pre-existing, unrelated" dismissal** (Task 4): the full suite intermittently failed with `RuntimeError: Event loop is closed` during asyncpg pool cleanup, reproducing only after enough prior DB-touching tests had run in the same process. Root cause: `app/db/session.py`'s module-level `_engine`/`_sessionmaker` globals outlived pytest-asyncio's per-test event loops even after `.dispose()`. Fixed by also nulling both globals in `db_session`/`clean_db` fixtures' `finally` blocks (`backend/tests/conftest.py`), forcing a fully fresh `AsyncEngine`/sessionmaker per test, not just a fresh pool тАФ extends the existing `test_db_session.py` precedent. **If a future milestone adds a new DB-touching fixture and this class of failure resurfaces, this is the fix pattern.**
- **`ai_platform/evaluation/`** is still README-only тАФ expected, lands in Milestone 8.
- `ChatWorkflow.log()`'s "chat turn complete" summary log line runs after `Workflow.run()`'s template method has already drained/closed the `execute()` generator, so by the time it fires, `workflow_ctx_var`/`conversation_id_ctx_var` are already reset to `None` (only `request_id` survives, from the broader HTTP-level ContextVar). This is inherited unchanged from Milestone 2's `Workflow.run()` ordering, not introduced this milestone, and doesn't affect this milestone's own logging acceptance criterion (`ToolExecutor`'s own log line fires mid-request with full context, verified above). Low-priority cleanup if it ever matters: either log a summary line from inside `execute()` before the `finally`, or restructure `Workflow.run()`'s template ordering тАФ needs a decision, not a quick patch, since `Workflow` is shared base infrastructure other future workflows also depend on.
- **`tool_executions` rows can be silently rolled back even after the tool genuinely ran.** `ToolExecutionRepository.record_execution()` only `flush()`es, by design тАФ the actual `commit()` happens once, at the end of `backend/app/api/chat.py`'s `event_stream()`, *after* Phase 2 (the response-generation LLM call) has also completed. If Phase 2 raises (e.g. `AIError` from a rate limit or a dropped connection) after a tool already executed successfully, the endpoint's exception handler calls `await db.rollback()`, which erases the `tool_executions` row for a tool call that *did* actually happen. For this milestone's only tool (`get_current_date`, side-effect-free) this is low-impact, but every future milestone's real tools inherit this same commit ordering тАФ a tool with a real side effect (e.g. posting a payment) would have the side effect occur with no durable audit trace surviving a later Phase 2 failure. Needs a deliberate decision (e.g. committing the audit write in its own transaction/session, or restructuring where commits happen in the request lifecycle), not a reflexive patch тАФ flagged here for whoever picks up Milestone 4/5's real tools.
- **`tool_executions.status` is an unconstrained `String(16)` column.** Only `"success"`/`"error"` are ever written by `ToolExecutor`, but nothing at the DB level enforces this тАФ no `CHECK` constraint, no native enum. Cheap to tighten later (the value set is small and closed) but not fixed now since it doesn't affect correctness today.

## 6. Do NOT Do

- Don't add `.eslintrc.json` back тАФ ESLint v9 flat config only.
- Don't add real finance tools yet тАФ that's Milestone 4/5. This milestone proves the tool-calling *mechanism* with one toy tool (`get_current_date`), nothing finance-specific.

- Don't let a tool failure raise all the way out of `ChatWorkflow` тАФ `ToolExecutor` must keep degrading failures to a `status="error"` outcome that Phase 2 can explain, never a crash.
- Don't route structured log fields through `logging.info(..., extra={...})` тАФ `JSONFormatter` only reads the dedicated ContextVars. See known-issue note above about `ChatWorkflow.log()`'s summary line before assuming every log call automatically has full context.
- Don't trust an implementer/reviewer's "pre-existing, unrelated failure" claim without independently reproducing it тАФ this exact milestone caught two real bugs (Task 4's fixture-disposal bug, Task 14's stale-placeholder-on-error bug) that were initially mischaracterized or under-scoped, both fixed only because the controller verified directly rather than accepting the claim.
