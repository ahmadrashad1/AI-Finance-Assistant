## 4. Decisions Made

- **`ExecutionPlanner` is a pure, synchronous resolver, not a `run()`-style mini-executor.** It exposes only `resolve_parameters(parameters, prior_outcomes) -> (resolved | None, error | None)` — no `ToolExecutor` dependency, no async, no internal loop. This was a deliberate refinement during design: an internal-loop design would have meant `ChatWorkflow` handing its whole tool-call loop to `ExecutionPlanner`, which would have broken the existing per-step `tool_call` SSE streaming event that the frontend and tests depend on. Keeping resolution as a single pure function let `ChatWorkflow` keep owning its loop and call `resolve_parameters` once per step, immediately before that step's own execution.
- **`"$stepN.field"` is a literal string match, not a template language.** `N` is the 0-based index into the same turn's `tool_calls` list; `field` is a key in that step's result. No expressions, no nested paths, no arithmetic — deliberately minimal, matching this milestone's stated scope ("simple parameter piping").
- **The 5-tool-call cap degrades to a clarifying question, not an error.** `Plan` already had a three-branch contract (`clarification_needed` / `tool_calls` / `direct_answer`); an oversized plan reuses the first branch rather than introducing a new failure mode `ChatWorkflow` would need to handle separately.
- **A resolution failure on one step degrades only that step (`continue`), never aborts the whole plan.** Verified deliberately by this session's live check (turn 2 of the "those" scenario, Task 27's own eval test) and by Task 20's reviewer tracing the source directly — the loop announces/attempts every planned step via its `tool_call` SSE event regardless of whether an earlier step's resolution succeeded.
- **`get_customer` is a deliberately narrow tool — code lookup only, no balance, no invoices** — kept as a strict subset of `get_customer_balance`'s result, specifically so the planner has a clean, single-purpose tool to pipe into a later call that needs a business code but was only given a company name. Its own `ToolSpec.description` explicitly tells the planner not to use it for balance questions.
- **Turn summaries are fetched before planning and only written after tool execution, so a turn can never see a summary of itself.** `ChatWorkflow.execute`'s ordering (`get_recent_turn_summaries` → `add_message` → `create_plan` → tool-call loop → `record_turn_summary`) makes this true by construction, not by a defensive check — confirmed by Task 24's reviewer tracing the exact line ordering.
- **`TurnSummaryModel.created_at` needed a client-side, timezone-aware default (`datetime.now(UTC)`), not `server_default=func.now()` alone.** PostgreSQL's `now()` is transaction-scoped (constant across every statement in one open transaction) — a real bug the implementer found and fixed during Task 22, since this milestone's own test records 4 turn summaries in a loop within one transaction before committing, and would have gotten 4 identical, untie-breakable timestamps otherwise. The first fix attempt used a naive `datetime.now`, which task review caught as inconsistent with this codebase's established `datetime.now(UTC)` convention (`get_current_date.py`) on a column typed `DateTime(timezone=True)`; corrected in a follow-up fix commit (`23185f7`).
- **Reasoning-query answers (e.g. payment prioritization) are handled entirely in the prompt layer, not in code.** No new "reasoning" tool or service method exists — the planner is taught (1.3.0) to plan `get_vendor_invoices` + `get_cash_position` together for this question shape, and Phase 2 is taught (1.4.0) to ground its recommendation strictly in the returned figures. Keeps the two-phase architecture and CLAUDE.md's "no business logic in the LLM, no keyword matching in code" rule intact for a genuinely new question shape.
- Executed directly on `master`, no feature branch — same policy as Milestones 1-6.

## 6. Do NOT Do

- **Don't add a `run()`-style method to `ExecutionPlanner`.** It's deliberately a pure, synchronous `resolve_parameters` function so `ChatWorkflow` keeps owning its own tool-call loop and its existing per-step `tool_call` SSE event stays intact. See §4.
- **Don't extend `"$stepN.field"` into a richer template/expression language without re-confirming this milestone's scope decision.** It is deliberately a literal, fully-anchored string match (`^\$step(\d+)\.(\w+)$`) — no nested paths, no arithmetic.
- **Don't raise an exception when a plan exceeds the 5-tool-call cap.** Return `Plan(clarification_needed=...)` — reuse the existing three-branch contract, don't add a fourth failure mode.
- **Don't let a single failed/unresolvable step abort the rest of a multi-step plan.** Degrade that one step to an error outcome via `continue` and proceed — proven both by unit test and by this session's live check.
- **Don't use `server_default=func.now()` alone when multiple rows might be inserted within one open transaction and their relative order matters.** PostgreSQL's `now()` is transaction-scoped, not per-statement — use a client-side, timezone-aware default (`datetime.now(UTC)`, matching `get_current_date.py`'s convention) instead, or in addition. See §4's `TurnSummaryModel.created_at` finding.
- **Don't fetch or populate turn-summary memory in the wrong order.** Recent turn summaries must be fetched *before* planning and turn-summary population must happen *after* tool execution — reversing either lets a turn see or record a summary of itself.
- **Don't call a tool handler with only `(params)`** — every handler takes `(params, context: ToolContext)`, even if it ignores `context`.
- **Don't add a new list-returning tool that dumps its full result into the Phase-2 prompt without going through `cap_result_for_prompt`** (Milestone 6, still true).
- **Don't assume a tool's `description` field reaches Phase 2** — Phase 1 (planning) reads `ToolSpec.description`; Phase 2 only ever sees `{tool, status, result, error}` (Milestone 6, still true, still relevant to §5's carried-forward AP-framing item).
- **Don't add a second `AsyncSession` to make `tool_executions` durable** (Milestone 6, still true).
- **Don't reimplement invoice- or vendor-invoice-status derivation logic anywhere else** — `compute_invoice_status`/`compute_vendor_invoice_status` remain the one source of truth for their respective sides.
- **Don't reuse `PaymentRepository.record_payment()` or `VendorPaymentRepository.record_payment()` behind a real, user-facing tool without first adding the validation from Milestone 4 HANDOFF §5** — still unaddressed on both sides now, still a must-fix before any write-tool.
- **Don't run the full pytest suite and then assume the seeded dev database is still populated** — `clean_db` truncates the same dev database the app and any manual/live check use; reseed after tests, before any live verification (this session hit this again and reseeded twice, matching Milestone 6's documented procedure).
- **Don't assume Docker Desktop is running** — it had exited between sessions this time; check (`docker compose ps`) and start it if needed before any DB-dependent work.
- **Don't add `.eslintrc.json` back** — ESLint v9 flat config only.
- **Don't trust an implementer/reviewer's "pre-existing, unrelated failure" claim without independently reproducing it.**
- **Don't assume `make` works on this machine without checking.**
